= AgnosticV Integration Guide
:navtitle: AgnosticV Integration Guide
:source-highlighter: rouge

Reference for lab authors who want to add LiteMaaS virtual keys to their own AgnosticV catalogs.

== Standard catalog configuration

Add the following to your catalog's `common.yaml`:

[source,yaml]
----
# Standard AgnosticV includes — required in every catalog
#include /includes/agd-v2-mapping.yaml
#include /includes/terms-of-service.yaml
#include /includes/parameters/purpose.yaml

# LiteMaaS connection (URL + master key)
#include /includes/secrets/litemaas-master_api.yaml
# Passes requester_email and catalog_item_name to LiteLLM for usage tracking
#include /includes/parameters/litellm_metadata.yaml

# Collection requirement
requirements_content:
  collections:
    - name: "https://github.com/rhpds/rhpds.litellm_virtual_keys.git"
      type: git
      version: main

# Workloads — run litellm keys BEFORE Showroom so credentials are baked in
workloads:
  - rhpds.litellm_virtual_keys.ocp4_workload_litellm_virtual_keys

# Run on destroy to delete the key
remove_workloads:
  - rhpds.litellm_virtual_keys.ocp4_workload_litellm_virtual_keys

# Hardcode the models your lab uses
ocp4_workload_litellm_virtual_keys_models:
  - "granite-3-2-8b-instruct"
  - "nomic-embed-text-v1-5"

# Enable Docling for PDF/DOCX → Markdown conversion
ocp4_workload_litellm_virtual_keys_include_docling: false

ocp4_workload_litellm_virtual_keys_duration: "7d"
----

== Enable Docling

Set `ocp4_workload_litellm_virtual_keys_include_docling: true` in your catalog to add Docling support.

When enabled, the role adds `granite-docling-258m` to the key and stores two additional user data variables:

* `{litellm_docling_endpoint}` — the `/docling/v1alpha/convert/source` endpoint URL
* `{litellm_docling_model}` — `granite-docling-258m`

Use `ifdef::litellm_docling_endpoint[]` in your `.adoc` files to guard Docling content — the attribute is only defined when Docling is enabled.

For curl and Python usage examples, see xref:07-module-05-docling.adoc[Module 5: Document Conversion].

== Multi-user workshops

[source,yaml]
----
__meta__:
  catalog:
    multiuser: true
    parameters:
      - name: num_users
        formLabel: Number of Users
        formGroup: Lab Configuration
        openAPIV3Schema:
          type: integer
          default: 1
          minimum: 1
          maximum: 30
----

TIP: `{litellm_virtual_key}` automatically resolves to each logged-in user's own key — no changes needed in your content.

== User data variables

The role stores these variables in AgnosticD user data. Showroom injects them as AsciiDoc attributes — use `{variable}` in your content and add `subs="attributes+"` to source blocks.

[cols="2,3",options="header"]
|===
|Variable |Value

|`litellm_api_base_url`
|Full `/v1` base URL (e.g. `https://litellm-prod.apps.maas.redhatworkshops.io/v1`)

|`litellm_virtual_key`
|The `sk-...` virtual key scoped to the selected models

|`litellm_available_models`
|List of model names the key can access (array)

|`litellm_available_models_list`
|Same as above, comma-separated string for display — includes `granite-docling-258m` when Docling is enabled

|`litellm_key_duration`
|Key validity period (e.g. `7d`)

|`litellm_tokenize_endpoint`
|`{litellm_api_base_url}/tokenize` — count tokens before sending to model

|`litellm_embeddings_endpoint`
|`{litellm_api_base_url}/embeddings` — generate text embeddings

|`litellm_docling_endpoint`
|Docling conversion URL — *only set* when `include_docling: true`

|`litellm_docling_model`
|`granite-docling-258m` — only set when Docling is enabled
|===

=== Verify your key and list available models

Use `litellm_api_base_url` and `litellm_virtual_key` to call the models endpoint — this is how users confirm their key works:

[source,bash,subs="attributes+"]
----
curl {litellm_api_base_url}/models \
  -H "Authorization: Bearer {litellm_virtual_key}" \
  -s | python3 -c "import sys,json; [print(m['id']) for m in json.load(sys.stdin)['data']]"
----

Or in Python:

[source,python,subs="attributes+"]
----
from openai import OpenAI

client = OpenAI(base_url="{litellm_api_base_url}", api_key="{litellm_virtual_key}")
models = client.models.list()
for m in models.data:
    print(m.id)
----

== Optional — model checkboxes in order form

Instead of hardcoding, let users pick models at order time:

[source,yaml]
----
ocp4_workload_litellm_virtual_keys_models: >-
  {{
    (model_granite_8b  | default(true)  | bool | ternary(['granite-3-2-8b-instruct'], [])) +
    (model_nomic_embed | default(false) | bool | ternary(['nomic-embed-text-v1-5'], [])) +
    (model_deepseek    | default(false) | bool | ternary(['deepseek-r1-distill-qwen-14b'], []))
  }}

__meta__:
  catalog:
    parameters:
      - name: model_granite_8b
        formLabel: Granite 3.2 8B Instruct
        formGroup: Models
        openAPIV3Schema:
          type: boolean
          default: true
      # repeat for each model...
----
