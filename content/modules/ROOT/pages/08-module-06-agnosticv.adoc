= Module 6: AgnosticV Integration
:navtitle: Module 6: AgnosticV Integration
:source-highlighter: rouge

This module is for lab authors who want to add LiteMaaS virtual keys to their own AgnosticV catalogs.
It shows exactly how the virtual key workload is configured, how model selection works, and how the credentials flow into Showroom.

== Learning objectives

By the end of this module, you'll be able to:

* Add the `ocp4_workload_litellm_virtual_keys` role to a catalog
* Configure model selection with checkboxes
* Set up multi-user workshops with per-user keys via `num_users`
* Enable Docling for document conversion labs
* Use the injected variables in Showroom content

== How variables flow

When a user provisions a lab, the role:

. Creates a virtual key scoped to the selected models
. Stores the key and endpoint in AgnosticD user data
. Showroom reads the user data and injects values as AsciiDoc attributes
. Your lab content uses `{litellm_virtual_key}`, `{litellm_api_base_url}`, etc.

[source]
----
AgnosticV catalog → AgnosticD role → Virtual key created → User data stored
                                                                ↓
                                        Showroom reads user data → injects as attributes
                                                                ↓
                                              Lab content: {litellm_virtual_key}
----

== Exercise 1: Standard catalog configuration

Most catalogs hardcode the model list directly. Pick the models your lab needs and list them:

[source,yaml]
----
# Standard AgnosticV includes — required in every catalog
#include /includes/agd-v2-mapping.yaml
#include /includes/terms-of-service.yaml
#include /includes/parameters/purpose.yaml

# LiteMaaS connection (URL + master key)
#include /includes/secrets/litemaas-master_api.yaml
# Passes requester_email and catalog_item_name to LiteLLM for usage tracking
#include /includes/parameters/litellm_metadata.yaml

# Collection requirement
requirements_content:
  collections:
    - name: "https://github.com/rhpds/rhpds.litellm_virtual_keys.git"
      type: git
      version: main

# Workloads — run litellm keys BEFORE Showroom so credentials are baked in
workloads:
  - rhpds.litellm_virtual_keys.ocp4_workload_litellm_virtual_keys

# Run on destroy to delete the key
remove_workloads:
  - rhpds.litellm_virtual_keys.ocp4_workload_litellm_virtual_keys

# Hardcode the models your lab uses
ocp4_workload_litellm_virtual_keys_models:
  - "granite-3-2-8b-instruct"
  - "nomic-embed-text-v1-5"

# Enable Docling if your lab needs document conversion
ocp4_workload_litellm_virtual_keys_include_docling: false

ocp4_workload_litellm_virtual_keys_duration: "7d"
----

== Exercise 3: Multi-user Showroom

For instructor-led workshops where each attendee needs their own key, add `num_users` to the catalog.
The role creates one key per user (e.g. `virtkey-{guid}-user1`, `user2`, `user3`).
Showroom then injects the *correct key for each logged-in user* automatically via `{litellm_virtual_key}`.

[source,yaml]
----
# Each user gets their own scoped virtual key
__meta__:
  catalog:
    multiuser: true
    parameters:
      - name: num_users
        formLabel: Number of Users
        formGroup: Lab Configuration
        openAPIV3Schema:
          type: integer
          default: 1
          minimum: 1
          maximum: 30
----

TIP: No changes needed in the role configuration or Showroom content.
`{litellm_virtual_key}` in your `.adoc` files automatically resolves to *the requesting user's key*.
User 1 sees their key, user 2 sees theirs — same template, personalised per login.

== Exercise 4: Enable Docling

Add document conversion support:

[source,yaml]
----
# Enable Docling — adds granite-docling-258m to the key
# and sets litellm_docling_endpoint in user data
ocp4_workload_litellm_virtual_keys_include_docling: "{{ enable_docling | default(false) | bool }}"

__meta__:
  catalog:
    parameters:
      - name: enable_docling
        formLabel: Document Conversion (Docling)
        formGroup: Models
        description: |
          Enable Docling for PDF/DOCX to Markdown conversion.
          Uses the /docling/v1alpha/convert/source endpoint.
        openAPIV3Schema:
          type: boolean
          default: false
----

== Available user data variables

These variables are stored in AgnosticD user data and injected into Showroom:

[cols="2,3",options="header"]
|===
|Variable |Value

|`litellm_api_base_url`
|Full `/v1` base URL (e.g., `https://litellm-prod.apps.../v1`)

|`litellm_virtual_key`
|The `sk-...` virtual key

|`litellm_available_models`
|List of model names (array)

|`litellm_available_models_list`
|Comma-separated model names (string, for display)

|`litellm_key_duration`
|Key validity (e.g., `7d`)

|`litellm_tokenize_endpoint`
|`/v1/tokenize` endpoint URL

|`litellm_embeddings_endpoint`
|`/v1/embeddings` endpoint URL

|`litellm_docling_endpoint`
|Docling endpoint URL — *only defined* when `include_docling: true`. Use `ifdef::litellm_docling_endpoint[]` to guard content.

|`litellm_docling_model`
|Docling model name — only defined when docling is enabled
|===

== Per-model Showroom conditional flags

The role also stores per-model flags in user data — but *only when that model is in the key*.
This means `ifdef::flag[]` correctly hides content when the model was not selected.

[cols="2,3",options="header"]
|===
|Attribute |Defined when...

|`litellm_has_embeddings`
|`nomic-embed-text-v1-5` is in the key

|`litellm_has_codellama`
|`codellama-7b-instruct` is in the key

|`litellm_has_llama_guard`
|`Llama-Guard-3-1B` is in the key

|`litellm_has_deepseek`
|`deepseek-r1-distill-qwen-14b` is in the key

|`litellm_has_phi4`
|`microsoft-phi-4` is in the key

|`litellm_has_qwen3`
|`qwen3-14b` is in the key

|`litellm_has_gpt_oss`
|`gpt-oss-120b` is in the key

|`litellm_has_llama_scout`
|`llama-scout-17b` is in the key
|===

Use these in your `.adoc` files to show content only when relevant:

[source,asciidoc]
----
\ifdef::litellm_has_embeddings[]
== Embeddings Exercise

[source,bash,subs="attributes+"]
 curl {litellm_api_base_url}/embeddings \
   -H "Authorization: Bearer {litellm_virtual_key}" \
   -d '{"model":"nomic-embed-text-v1-5","input":"Hello world"}'

\endif::litellm_has_embeddings[]

\ifndef::litellm_has_embeddings[]
NOTE: This module requires Nomic Embed v1.5, which is not enabled in your key.
\endif::litellm_has_embeddings[]
----

== Using variables in Showroom AsciiDoc

In your Showroom `.adoc` files, reference variables as attributes:

[source,asciidoc]
----
Your API base URL: `{litellm_api_base_url}`

Your virtual key: `{litellm_virtual_key}`

Run this to test:

[source,bash,subs="attributes+"]
 curl {litellm_api_base_url}/models \
   -H "Authorization: Bearer {litellm_virtual_key}"
----

IMPORTANT: Always add `subs="attributes+"` to code blocks where you use `{variable}` substitution.
